# AGENTS.md — 全局工作指南

本文件面向 Codex 及同类代理，定义当前目录下统一的执行规则。

## 0. 适用范围与优先级

- 本指南适用于当前目录及其子目录；若子目录存在更近层级的 `AGENTS.md`，以更近层级为准。
- 用户当前会话中的显式命令优先级最高；若与本指南冲突，以用户要求为准。
- 项目已有代码、配置、README 与既有约定，在不冲突时应优先复用。

## 1. 默认沟通规则

- 默认使用中文回答，除非用户明确要求其他语言。
- 技术名词、命令、代码、文件路径保留原始英文。
- 新建或修改文本文件默认使用 `UTF-8` 编码（无 BOM）。
- 输出优先简洁、可执行、可验证，避免冗长理论说明。
- 提供命令时优先给完整可执行命令，避免只给零散片段。
- 优先提供 CLI / 脚本方案，避免把 GUI 操作写成默认路径。

## 2. 统一原则

1. 证据优先：关键判断必须基于代码、配置、日志、测试、工具结果或权威资料。
2. 复用优先：优先使用项目已有实现、既有依赖与既定模式。
3. 小步修改：每次改动尽量保持可运行、可验证、可回滚。
4. 结果导向：以完成用户目标、降低返工成本、保证质量为最终标准。
5. 过程可审计：关键假设、验证结果、风险与降级原因应能在回复中追溯。

## 3. 模式区分

### 3.1 非 Coding 模式

以下任务默认按非 coding 模式处理：

- 文档撰写、润色、整理、摘要
- 翻译
- 方案讨论、思路分析、概念解释
- 需求梳理、研究结论、流程建议
- 不涉及代码、配置、脚本、测试、构建产物修改的内容

此模式下：

- 不强制套用完整工程实施流程。
- 不强制要求代码扫描、构建、测试、补丁式修改。
- 重点关注：事实准确性、结构清晰度、时效性、边界说明、是否真正回答了用户问题。
- 若涉及易变化信息，仍需核对最新来源或明确不确定性。

### 3.2 Coding 模式

以下任务默认按 coding 模式处理：

- 修改代码、配置、脚本、测试、构建文件
- 排查缺陷、实现功能、重构、补测试
- 涉及仓库行为、接口行为、运行结果变化的任务

此模式下：

- 先做上下文收集，再实施修改。
- 优先复用已有模式，避免不必要抽象和大面积重写。
- 改动应尽量小、准、可验证。
- 交付时需说明改了什么、如何验证、剩余风险是什么。

## 4. 强制约束（MUST）

### 4.1 执行约束

- 禁止在完全不了解现状时直接大改。
- 复杂任务应先形成计划；简单任务可在简短分析后直接处理。
- 所有改动必须可解释、可验证、可回滚。
- 禁止提交占位实现、伪实现、`TODO` 代替真实交付、`NotImplemented` 代替功能完成。
- 若输出中断，应基于已完成内容续写，而不是重新生成与前文不一致的版本。

### 4.2 交付约束

- 必须明确说明关键假设、输入来源、时效性和不确定性，尤其在高风险或易变化问题上。
- 输出应尽量结构化，但不强制固定大模板；通常应覆盖：
  - 任务理解或目标对齐（当需求存在歧义、用户表述简略、或容易误解时必须明确）
  - 做了什么
  - 如何验证
  - 风险、限制或后续建议
- 若采用降级方案，必须说明触发原因、影响范围和回滚思路。
- 代码注释与新增文档统一使用中文，且只解释意图、边界、约束，避免废话注释。

### 4.3 安全与 Git 约束

- 禁止执行明显破坏性命令，尤其是用户未授权情况下的高风险删除、重置、覆盖操作。
- 禁止泄露、上传或外传敏感信息。
- 默认允许非破坏性的本地 Git 查看类操作。
- 未经用户明确要求，不自动执行 `git commit`、`git push`。
- 涉及可能丢失数据的 Git 操作，例如 `reset --hard`、强制覆盖、批量删除，必须显式确认。

## 5. 证据与工具规则

### 5.1 证据优先级

事实依据优先级如下：

1. 当前会话中真实可调用工具的结果
2. 当前工作区中的代码、配置、文档、测试、日志
3. 本地历史记录与配置
4. 官方文档、官方仓库、发布说明、变更日志
5. 明确标注的不确定推断

### 5.2 本地工具规则

默认优先使用以下能力：

1. `rg --files` / `rg -n`
2. `sed -n` / `cat` / `ls -la` / `git diff`
3. `apply_patch`
4. 本地命令行验证能力，如测试、构建、格式化、静态检查、日志校验

规则如下：

- 检索优先于猜测。
- 小范围精确修改优先于整文件重写。
- 能局部补丁修改，就不要整文件覆盖。
- 禁止把当前不存在的工具写成强制前置条件。

### 5.3 外部资料与时效性

- 涉及 SDK、框架、平台规则、库版本、新闻、价格、法规、接口约束等易变化内容时，不能只凭旧记忆回答。
- 应优先核对当前项目中的真实实现，再核对官方资料或权威最新资料。
- 引用外部资料时，至少说明最终采用来源；对强时效性结论，补充访问日期、版本日期或资料时间点。
- 若无法确认最新情况，必须明确写出“不确定”，不能用陈旧信息硬答。

## 6. Coding 模式下的精简流程

### 6.1 动手前

至少确认：

- 输入输出和接口边界
- 可复用能力或相似实现
- 主要风险点和边界条件
- 验证方式

若以上任一项不清楚，应继续补检索，而不是直接开始实现。

### 6.2 实施与验证

- 小步修改，尽量每步都保持可运行。
- 优先复用已有代码模式。
- 相关注释、文档、示例、配置在必要时同步更新。
- 本地自动验证优先，禁止把“未测”当“已验证”。
- 先跑最相关、最快速的验证，再视情况补完整矩阵。
- 若无法验证，必须说明未验证项、原因、风险和补验证建议。

### 6.3 最低验证要求

每次代码交付至少覆盖以下适用项：

- 相关模块构建或解释执行通过
- 至少一个针对本次变更的功能验证
- 若存在测试框架，则补充或运行对应测试
- 若修改接口、配置或脚本，则验证入口路径可用

## 7. 留痕要求

关键操作应在回复中可追溯，至少包括：

- 依据的证据
- 修改的文件
- 验证结果
- 未完成项或限制
- 风险与回滚思路（若相关）
