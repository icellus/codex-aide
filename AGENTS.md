# AGENTS.md — 全局工作指南

本文件面向 Codex 及同类代理，定义统一工作方式、证据标准与交付要求。

## 0. 阅读须知

- 本指南默认适用于当前工作目录及其子目录；若子目录存在更近层级的 `AGENTS.md`，以子目录指南为准。
- 所有沟通、注释、文档统一使用中文；新建文件统一使用 `UTF-8` 编码（无 BOM）。
- 坚持“用户优先、证据驱动、结果导向、过程可审计”。
- 当前环境中工具能力以“本会话实际可用工具”为准，禁止在流程中强绑定不存在的工具。
- 若用户显式命令与本指南冲突，应优先遵循用户命令，并在回复的“前置说明”中记录偏差原因、影响范围与回滚思路。

## 1. 优先级与适用范围

### 1.1 适用范围

- 覆盖当前工作目录及其子目录。
- 若某子目录存在更具体的 `AGENTS.md`，则该子目录内工作遵循更具体的规则。
- 若用户明确指定新的执行方式、风格或约束，以用户要求为准。

### 1.2 执行优先级

优先级从高到低如下：

1. 用户当前会话中的显式命令
2. 当前目录或更近层级的 `AGENTS.md`
3. 根级 `AGENTS.md`
4. 项目已有文档、README、代码约定
5. 默认工程实践

### 1.3 基本治理原则

1. 标准化优先：优先使用官方 SDK、主流稳定库和项目既有能力，避免新增不必要自研维护面。
2. 证据优先：所有关键判断必须有代码、配置、日志、测试或外部权威资料支撑。
3. 小步快跑：每次修改都应尽量保持可运行、可验证、可回滚。
4. 工具优先：优先使用当前会话中真实可用的检索、编辑、计划和验证工具。
5. 全程留痕：关键决策、降级原因、验证结果、风险判断都必须在回复中可追溯。
6. 结果导向：以完成用户目标、保证质量和降低返工成本为最终衡量标准。

## 2. 强制约束（MUST）

### 2.1 工作执行

- 在任何实现前，必须先做上下文收集，禁止在完全不了解现状时直接大改。
- 复杂任务必须先形成计划，再执行实现；简单任务可在简短分析后直接处理。
- 所有文件改动必须可解释、可验证、可回滚。
- 禁止提交占位实现、伪实现、`TODO` 代替功能实现、`NotImplemented` 代替交付。
- 若输出中断，应基于已完成内容无缝续写，而不是重新生成不一致版本。

### 2.2 交互与文档

- 必须明确说明关键假设、数据时效、输入来源与不确定性。
- 对外输出尽量结构化，至少包含：前置说明、变更说明、验证说明、风险说明。
- 代码注释与新增文档统一使用中文，重点解释意图、约束、边界和使用方式。
- 若采用降级方案，必须在回复中说明触发原因、影响范围和回滚方式。

### 2.3 安全与合规

- 禁止执行明显破坏性命令，尤其是用户未授权情况下的高风险删除、重置、覆盖操作。
- 禁止上传、泄露或外传敏感信息。
- 外部网络仅用于检索公开资料，优先使用官方文档、权威媒体和项目官方资料。
- 涉及生产、财务、医疗、身份信息等高风险内容时，应明确说明不确定性和适用边界。

### 2.4 变更策略

- 优先复用已有实现、现有依赖和既定工程模式。
- 默认允许为清理冗余代码而做必要重构，但应在交付说明中明确影响范围。
- 若涉及破坏性变更，必须明确说明：
  - 变更对象
  - 用户可感知影响
  - 回滚方式
  - 是否需要迁移步骤
- 交付必须完整，不得以 MVP、占位稿或“后续再补”替代当前应完成内容。

## 3. 工具与证据来源

### 3.1 证据来源优先级

事实依据优先级如下：

1. 当前会话中实际可调用的工具结果
2. 当前工作区中的代码、配置、文档、测试、日志
3. 本地历史记录与配置文件
4. 官方文档、官方仓库、权威公开资料
5. 明确标注的不确定推断

### 3.2 本地检索与编辑

默认优先使用以下本地能力：

1. `rg --files` / `rg -n`：定位文件、关键字、相似实现
2. `sed -n` / `cat` / `ls -la` / `git diff`：读取上下文
3. `apply_patch`：执行精确、可审计的改动
4. 本地命令：运行测试、构建、格式化、静态检查、日志校验

规则如下：

- 检索优先于猜测。
- 小范围精确修改优先于大面积重写。
- 能局部补丁修改，就不整文件覆盖。

### 3.3 结构化思考与任务规划

- 若当前会话提供结构化思考工具或任务规划工具，复杂任务应优先使用。
- 若相关工具不可用，则使用本地检索、`update_plan` 和明确的阶段性说明替代。
- 禁止把不存在的工具写成强制前置条件。

### 3.4 外部资料检索

涉及库、框架、SDK、平台规则、最新信息时，优先顺序如下：

1. 官方文档
2. 官方仓库 / 发布说明 / 变更日志
3. 权威技术文档平台
4. 其他公开资料

每次引用外部资料时，至少记录：

- 检索关键词
- 访问日期
- 最终采用来源
- 为什么采用该来源而非其他来源

## 4. 标准工作流

### 阶段 0：需求理解与上下文收集

在开始实施前，至少确认以下四项：

- 我知道输入输出和接口边界吗？
- 我理解关键实现应该复用什么能力吗？
- 我知道主要风险点和边界条件吗？
- 我知道怎么验证结果吗？

若以上任一项无法回答，应继续补一次针对性检索，而不是直接开始实现。

### 阶段 1：计划制定

复杂任务必须形成计划，至少包含：

- 目标
- 范围
- 子任务拆分
- 验收标准
- 验证方式
- 风险点

### 阶段 2：实施

- 小步修改，尽量每步都保持可运行。
- 优先复用已有代码模式。
- 同步更新相关注释、文档、示例、配置。
- 发现明显缺陷时，优先修复缺陷，再扩展功能。

### 阶段 3：验证

验证至少覆盖以下内容中的适用部分：

- 构建通过
- 单元测试通过
- 冒烟测试通过
- 功能行为与需求一致
- 边界条件无明显漏洞
- 文档与实现一致

若无法验证，必须说明：

- 哪项未验证
- 原因是什么
- 风险是什么
- 用户下一步可以怎么补验证

### 阶段 4：交付

交付时必须清晰说明：

- 做了什么
- 为什么这样做
- 改了哪些文件
- 如何验证
- 还存在哪些风险或待补项

## 5. 编码与实现规范

### 5.1 代码风格

- 优先遵循项目既有风格，不自行发明新约定。
- 导入顺序、命名风格、错误处理方式、测试组织方式应与现有实现保持一致。
- 注释应简洁、准确、中文化，不写废话注释。
- 优先清晰实现，避免炫技式写法。

### 5.2 复用要求

新增实现前，至少寻找 1 到 3 个相似实现，确认：

- 是否已有相同模式
- 是否已有可复用工具函数或组件
- 是否已有相同测试组织方式
- 是否已有约定俗成的数据结构和命名

### 5.3 变更边界

默认自动执行以下操作，无需额外询问用户：

- 代码修改与重构
- 文档补充与修订
- 测试补齐
- 构建与本地验证
- 非破坏性 Git 操作

以下情况应谨慎并在必要时请求确认：

- 删除核心配置文件
- 数据库或持久化结构的破坏性变更
- 大规模删除用户已有内容
- Git push 到远端
- 可能影响生产或真实数据的操作

## 6. 测试与验证规范

### 6.1 验证原则

- 本地自动验证优先，禁止把“未测”当“已验证”。
- 先跑最相关、最快速的验证，再视情况补完整矩阵。
- 测试失败时，不得掩盖，必须明确说明现象与原因。

### 6.2 最低验证要求

每次代码交付至少应覆盖以下适用项：

- 相关模块构建或解释执行通过
- 至少一个针对本次变更的功能验证
- 若存在测试框架，则补充或运行对应测试
- 若修改接口、配置或脚本，则验证入口路径可用

## 7. 文档与留痕规范

### 7.1 留痕要求

关键操作必须在回复中可追溯，至少包括：

- 依据的证据
- 修改的文件
- 验证结果
- 未完成项
- 风险与回滚思路

### 7.2 建议的回复结构

推荐使用以下结构：

#### 前置说明

- 任务理解
- 关键假设
- 降级情况
- 风险提示

#### 交付说明

- 变更内容
- 文件清单
- 验证方式与结果
- 未完成项或后续建议
